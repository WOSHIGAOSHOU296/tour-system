## 二、分层架构（答辩核心）

                  │
                  ▼

## 三、数据库设计（核心亮点，建议重点讲）

### ER 关系图

```
users(用户) ──┬── user_role(关联) ──── roles(角色)
               │
               ├── strategies(攻略) ──── attractions(景点)
               ├── messages(留言) ────── attractions(景点)
               ├── forum_posts(帖子) ──── post_comments(评论)
               ├── trips(行程)
               ├── collections(收藏)
               └── browse_records(浏览记录)

cities(城市) ──── attractions(景点) ──┬── foods(美食)
                                     └── hotels(酒店)

景点管理员 ──── announcements(公告) ──── attractions(景点)
```

### 关键表设计亮点

**1. 用户角色体系（RBAC 模型）**：用三张表实现多对多关系

```sql
-- roles 表：定义角色
INSERT INTO roles VALUES (1, '游客'), (2, '普通用户'), (3, '景区管理员'), (4, '平台管理员');

-- user_role 表：用户和角色的关联
INSERT INTO user_role (user_id, role_id) VALUES (1, 4);  -- 用户1 = 平台管理员
```

这样设计的好处是：如果将来需要增加角色（如"内容审核员"），只需在 roles 表加一行，无需改代码。

**2. 收藏表的多态设计**：用 `collect_type` 字段区分不同资源类型
```sql
collect_type = 1 → 景点
collect_type = 2 → 攻略  
collect_type = 3 → 帖子
collect_type = 4 → 行程
```
一张表实现对四种资源的收藏，避免建四张表。

**3. 浏览记录表**：同样的多态设计模式
```sql
browse_type = 1 → 景点
browse_type = 2 → 攻略
browse_type = 3 → 帖子
```

**4. 景点浏览量自动统计**：每次查看详情时 `view_count + 1`
```sql
UPDATE attractions SET view_count = view_count + 1 WHERE scenic_id = ?;
```

## 四、核心代码讲解（答辩关键）

### 4.1 数据库连接工具类 (DBUtil.java)

```java
public class DBUtil {
    // JDBC 四大参数
    private static final String DRIVER = "com.mysql.jdbc.Driver";
    private static final String URL = "jdbc:mysql://localhost:3306/tour_db?...";
    private static final String USER = "root";
    private static final String PASSWORD = "***";

    // 静态代码块：类加载时执行一次，注册驱动
    static {
        Class.forName(DRIVER);
    }

    // 获取连接
    public static Connection getConnection() {
        return DriverManager.getConnection(URL, USER, PASSWORD);
    }

    // 释放资源（必须按 ResultSet → Statement → Connection 顺序关闭）
    public static void closeAll(ResultSet rs, Statement stmt, Connection conn) {
        // 每个资源独立 try-catch，确保一个关闭失败不影响其他的关闭
        if (rs != null) rs.close();
        if (stmt != null) stmt.close();
        if (conn != null) conn.close();
    }
}
```

**老师可能问**："为什么用 DriverManager 不用连接池？"  
→ 答：课程设计场景下数据量小、并发低，原生 JDBC 足够使用。生产环境中会用 Druid 连接池。

### 4.2 密码加密 (PasswordUtil.java)

```java
// 核心算法：MD5 + 随机盐值 (Salted Hash)
public static String encrypt(String password) {
    String salt = generateSalt();         // 1. 生成16位随机盐值
    String hash = md5(password + salt);   // 2. 密码+盐值 → MD5
    return salt + "#" + hash;             // 3. 存储格式: "salt#hash"
}

public static boolean verify(String password, String stored) {
    String salt = stored.split("#")[0];  // 1. 拆出盐值
    String hash = stored.split("#")[1];  // 2. 拆出哈希
    return md5(password + salt).equals(hash); // 3. 相同算法重新计算对比
}
```

**为什么加盐值？**  
→ 如果只用 `MD5("123456")`，黑客用彩虹表直接反查。加随机盐值后，即使两个用户密码相同，数据库里存的密文也完全不同。

### 4.3 Controller 的 action 分发机制

所有 Controller 共享同一个 URL 映射模式，通过 `?action=xxx` 参数区分不同操作：

```java
// web.xml 中只注册一个映射
<url-pattern>/forum</url-pattern>

// Controller 内部用 switch 分发
public void doPost(req, resp) {
    String action = req.getParameter("action");  // 从URL获取 ?action=xxx
    switch (action) {
        case "list":   list(req, resp);   break;  // 帖子列表
        case "detail": detail(req, resp); break;  // 帖子详情
        case "create": create(req, resp); break;  // 发帖页面
        case "doCreate": doCreate(req, resp); break; // 提交发帖
        ...
    }
}
```

**为什么这样设计？**  
→ 传统方式下每个功能需要单独的 Servlet 类，会导致几十个 Servlet 文件。用一个 `action` 参数分发，一个模块只需要一个 Servlet，代码集中好维护。这是 Servlet 3.1 规范中推荐的 REST 风格替代方案。

### 4.4 DAO 层的动态 SQL 构建

景点搜索功能需要同时支持城市筛选 + 关键词搜索 + 分页：

```java
// ScenicDao.findAll() — 动态拼接 SQL
StringBuilder sql = new StringBuilder(
    "SELECT a.*, c.city_name FROM attractions a " +
    "LEFT JOIN cities c ON a.city_id = c.city_id WHERE 1=1");

if (cityId != null && cityId > 0) {
    sql.append(" AND a.city_id = ?");    // 动态添加城市筛选条件
}
if (keyword != null && !keyword.isEmpty()) {
    sql.append(" AND (a.scenic_name LIKE ? OR a.introduce LIKE ?)"); // 模糊搜索
}
sql.append(" ORDER BY a.view_count DESC LIMIT ?, ?");  // 按热度排序 + 分页

// 使用 PreparedStatement 防止 SQL 注入（关键安全点）
stmt = conn.prepareStatement(sql.toString());
stmt.setLong(1, cityId);
stmt.setString(2, keyword);
```

**PreparedStatement 防 SQL 注入**：参数通过 `?` 占位符传入，不会被当作 SQL 代码执行。

### 4.5 浏览记录的多态联表查询

浏览记录表存的是 `target_id`（数字ID），展示时需要显示名称：

```sql
-- BrowseRecordDao.findByUser() — CASE WHEN + LEFT JOIN
SELECT b.*,
  CASE b.browse_type
    WHEN 1 THEN a.scenic_name      -- 类型1 → 查景点名
    WHEN 2 THEN s.title            -- 类型2 → 查攻略标题
    WHEN 3 THEN f.title            -- 类型3 → 查帖子标题
    ELSE '未知'
  END AS target_name
FROM browse_records b
LEFT JOIN attractions a ON b.browse_type=1 AND b.target_id=a.scenic_id
LEFT JOIN strategies s  ON b.browse_type=2 AND b.target_id=s.strategy_id
LEFT JOIN forum_posts f ON b.browse_type=3 AND b.target_id=f.post_id
WHERE b.user_id = ? ORDER BY b.browse_time DESC LIMIT ?, ?
```

**这是本项目的SQL难点**：用一条SQL的 `CASE WHEN` + 3个条件 `LEFT JOIN` 实现了对三种不同类型表的联合查询。

## 五、安全机制（三层防护）

### 第一层：LoginFilter — 登录拦截

```java
// 拦截路径: /forum, /trip, /strategy, /collection, /admin, /scenicAdmin
HttpSession session = req.getSession(false);
if (session == null || session.getAttribute("user") == null) {
    // 未登录 → 重定向到登录页，并记住来源URL
    resp.sendRedirect("/login.jsp?redirect=原请求地址");
    return;
}
// 检查账号状态
if (user.getStatus() == 0) {
    session.invalidate();  // 销毁会话
    resp.sendRedirect("/login.jsp?error=账号已被禁用");
    return;
}
chain.doFilter(request, response);  // 放行
```

### 第二层：AdminFilter — 平台管理员权限

```java
Long roleId = (Long) session.getAttribute("roleId");
if (roleId == null || roleId != 4) {
    // 不是平台管理员 → 拒绝访问
    resp.getWriter().write("<script>alert('无权限访问');history.back();</script>");
    return;
}
```

### 第三层：ScenicAdminFilter — 景区管理员权限

```java
Long roleId = (Long) session.getAttribute("roleId");
if (roleId == null || (roleId != 3 && roleId != 4)) {
    // 只有景区管理员(3)和平台管理员(4)可访问
    resp.getWriter().write("<script>alert('无权限访问');history.back();</script>");
    return;
}
```

### Filter 执行顺序（web.xml 中的注册顺序决定）

```
请求到达 → LoginFilter(检查登录) → AdminFilter(检查角色) → Controller → JSP
              ↓ 未登录                         ↓ 无权限
          重定向登录页                    弹出提示拒绝
```

### URL 映射安全注意事项

```xml
<!-- ⚠️ 必须同时配置精确路径和前缀路径，否则 /trip 裸路径无法被拦截 -->
<url-pattern>/trip</url-pattern>       <!-- 拦截 /trip?action=list -->
<url-pattern>/trip/*</url-pattern>     <!-- 拦截 /trip/xxx.jsp -->
```

## 六、数据流转完整示例（以"用户登录"为例）

```
1. 用户在 login.jsp 填写表单，点击"登录"
   ↓ POST /user?action=login, body: username=admin&password=123456

2. UserController.doPost() 接收请求
   → 解析 action=login
   → 调用 login(req, resp)

3. UserController.login()
   → userService.login("admin", "123456")

4. UserService.login()
   → userDao.findByUsername("admin")     // SELECT * FROM users WHERE username='admin'
   → 检查 user.status != 0               // 账号未被禁用
   → PasswordUtil.verify("123456", "salt#hash")  // 密码验证
   → userDao.findRoleIdByUserId(userId)  // SELECT role_id FROM user_role WHERE user_id=?
   → 返回 User 对象（密码字段已置 null）

5. 回到 UserController.login()
   → session.setAttribute("user", user)   // 存入会话
   → session.setAttribute("roleId", 4L)   // 存入角色
   → resp.sendRedirect("index.jsp")       // 重定向到首页

6. 浏览器收到 302 重定向
   → 自动请求 GET /index.jsp
   → header.jsp 读取 session 中的 user 对象
   → 导航栏显示 "平台管理员" + 退出按钮
```

## 八、老师可能问的问题及回答

**Q1：为什么选择 Servlet 而不是 Spring Boot？**
→ 答：课程设计的要求是掌握 Java Web 基础原理，Servlet 是 Java Web 的底层标准。先理解 Servlet 的请求-响应模型、Session 机制、Filter 过滤器链，对以后学习 Spring Boot 有很大帮助。Spring Boot 底层也是基于 Servlet 的。

**Q2：项目有什么不足？**
→ 答：(1) 密码加密目前用 MD5，应该升级到 BCrypt 更安全；(2) 数据库连接使用 DriverManager，高并发下应改用 Druid 连接池；(3) 前后端没有分离，大量 Java 代码混写在 JSP 中；(4) 没有实现文件上传功能导致景点图片只能手动放置。

**Q3：Session 和 Cookie 的关系？**
→ 答：用户登录后，服务器创建 Session 对象存储在内存中，Session ID 通过 Cookie（JSESSIONID）发送给浏览器。浏览器每次请求自动带上这个 Cookie，服务器根据它找到对应的 Session，从而识别用户身份。本项目设置了 Session 30 分钟超时。

**Q4：项目中有哪些数据库优化？**
→ 答：(1) 所有外键字段加了索引（如 `idx_city_id`），加速 JOIN 查询；(2) 分页查询使用 `LIMIT offset, pageSize`，避免一次加载全部数据；(3) 搜索使用了 `LIKE '%keyword%'` 模糊匹配，数据量大时可以考虑用 MySQL 全文索引。

**Q5：如果项目要上线，还需要做什么？**
→ 答：(1) 数据库密码从代码中提取到配置文件；(2) 添加文件上传功能供管理员上传景点图片；(3) 加入验证码防止暴力登录；(4) 使用 HTTPS 加密传输；(5) 添加日志系统记录异常和操作审计。
